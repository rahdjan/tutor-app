# SECURITY.md — Репетитор.Платформа

Итог предзапусковой ревизии безопасности (Фаза 11 брифа). Ниже — что уже
сделано и проверено, и что осталось сделать вручную перед реальным запуском
(раздел «Перед запуском» в конце).

## 1. Модель угроз и роли

Мультитенантная платформа: один сервер, много независимых репетиторов, у
каждого — свои ученики и контент. Роли: `TUTOR`, `STUDENT`, анонимный
посетитель. Главный риск — не внешний взлом, а **утечка между арендаторами**:
репетитор А увидел данные репетитора Б, или ученик увидел чужое задание.
Поэтому изоляция данных — приоритет №1 всего проекта (см. AGENTS.md, правило 1).

## 2. Изоляция данных

Единый слой доступа — `lib/access.ts`: `requireTutor()` / `requireStudent()`
читают сессию и либо возвращают её, либо делают redirect. Ни одна страница
кабинета и ни один server action не читают БД до вызова одной из этих функций.

Паттерн запросов, которого придерживается весь код:
- чтение — `findFirst({ where: { id, tutorId: session.user.id } })`: чужой id
  просто не находится → `notFound()` / `null`;
- изменение/удаление — `updateMany`/`deleteMany` с тем же условием: `count === 0`
  означает «не моё», а не «ошибка сервера»;
- фото решений — не отдаются напрямую из хранилища (см. §8), а через
  `app/api/solution-photo/[entryId]/route.ts`, который сам проверяет, что
  запись принадлежит либо ученику, либо его репетитору.

Аудит (2026-07-20) прошёл по всем **30+ server actions** (`app/actions/*.ts`)
и всем **3 route handlers** (`app/api/auth/[...all]`, `app/api/solution-photo`,
`app/tutor/tasks/export`) — везде обнаружен и гард роли, и фильтр владельца.
Регресс проверяется e2e-сценариями каждой фазы (двое репетиторов, ученик не
своего репетитора, прямые URL чужих сущностей — везде 404/redirect).

`proxy.ts` — только первый, оптимистичный рубеж (есть ли cookie сессии);
реальная проверка роли и владения — всегда на сервере, в `lib/access.ts` и
в самих actions. Так и задумано: proxy не может проверить роль без похода
в БД, а делать это на каждый запрос дорого.

## 3. Аутентификация и сессии

Better Auth (email + пароль, минимум 8 символов), сессии — в Postgres.
Роль (`role`) и привязка к репетитору (`tutorId`) помечены `input: false` в
конфиге пользователя (`lib/auth.ts`) — их нельзя передать с клиента ни через
форму, ни прямым запросом к `/api/auth/sign-up/email`; выставляет только
серверный код после проверки приглашения. Согласие на обработку данных
(и родительское для `under16`) проверяется в `databaseHooks.user.create.before`
— обход форм невозможен, хук отработает при любом способе создать аккаунта.

Cookie сессии: `HttpOnly`, `SameSite=Lax`, `Secure` в production (дефолты
Better Auth при https `BETTER_AUTH_URL`) — проверено вручную в DevTools.

## 4. Rate limiting

Два независимых уровня, потому что у Better Auth есть встроенный лимитер, но
он **не защищает наши формы**: при вызове `auth.api.signInEmail()` из server
action (а не через HTTP на `/api/auth/*`) лимитер Better Auth не участвует —
проверено чтением `better-auth/dist/api/dispatch.mjs` (вызов «как функции»
осознанно пропускает rate-limit хук).

**Уровень 1 — Better Auth, `storage: "database"`** (`lib/auth.ts`): защищает
прямые HTTP-запросы к `/api/auth/*` в обход наших форм (curl-перебор). Таблица
`rateLimit` в Postgres — `storage: "memory"` бесполезен на Vercel, где
одновременно живут несколько serverless-инстансов без общей памяти.
`/sign-in/email` — 10/мин, `/sign-up/email` — 10/час, `/get-session` не
лимитируется (клиентского auth-клиента в проекте нет, HTTP на этот путь не
идёт вовсе).

**Уровень 2 — собственный лимитер, `lib/rate-limit.ts`**: fixed-window на
отдельной таблице `app_rate_limit`, врезан прямо в server actions. Специально
**не переиспользует** таблицу Better Auth — та периодически чистит свои строки
по короткому cutoff (около минуты) и стёрла бы наши часовые/суточные окна.
Реализация — **без единой строки сырого SQL**: каждый шаг — одиночный условный
`updateMany` (атомарен на уровне Postgres) либо `create` под `@unique(key)` с
перехватом гонки. Fail-open при сбое БД (если Postgres недоступен, вход и так
не сработает — деградации защиты нет).

| Операция | Ключ | Лимит |
|---|---|---|
| Вход | IP и email отдельно | 10 / 10 мин, 10 / 15 мин |
| Регистрация репетитора | IP | 5 / час |
| Регистрация по приглашению | IP | 10 / час |
| ИИ-импорт (платный запрос к LLM) | userId | 10 / час **и** 40 / сутки |
| Загрузка фото решения | userId | 30 / час |
| Импорт задач из JSON | userId | 10 / час |
| Создание приглашения ученику | userId | 30 / час |

IP берётся из `x-real-ip` → `x-forwarded-for` (`lib/rate-limit.ts:clientIp`) —
на Vercel этот заголовок перезаписывается прокси реальным адресом клиента,
подделать нельзя.

## 5. SQL-инъекции

Весь доступ к БД — через типобезопасный Prisma Client API, сырых запросов нет
нигде в собственном коде (`$queryRaw`/`$executeRaw`/`*Unsafe` — 0 совпадений
при grep по `app/`, `lib/`, `prisma/`, вне `app/generated`). Инъекции
структурно невозможны: параметры всегда идут через типизированные аргументы
Prisma, а не конкатенацию строк. Это осознанная политика проекта — если в
будущем понадобится raw SQL, его нужно явно обсуждать и параметризовать через
`Prisma.sql` template tag, а не строковую конкатенацию.

## 6. XSS

Единственный источник произвольного HTML в приложении — рендер формул
LaTeX/Markdown (`lib/latex.ts`). Все 5 использований `dangerouslySetInnerHTML`
в проекте идут через `renderMathHtml()`: текст вне формул экранируется
(`escapeHtml` — `&`, `<`, `>`, `"`, `'`), KaTeX вызывается с
`throwOnError: false` (сломанная формула не роняет страницу, показывается как
код). Правило для будущего кода: новый пользовательский текст в HTML — только
через этот модуль, никогда напрямую.

## 7. Security-заголовки (`next.config.ts`)

- `Content-Security-Policy` — построена под фактический состав страницы:
  сторонних скриптов и доменов нет, KaTeX бандлится, фото решений отдаёт
  только `/api/solution-photo`. `'unsafe-inline'` в `script-src`/`style-src` —
  вынужденный компромисс App Router (инлайн-бутстрап RSC и стили KaTeX/Next
  без nonce); это второй рубеж поверх экранирования из §6, а не единственная
  защита. Апгрейд до nonce-based CSP через `proxy.ts` — возможное улучшение
  после запуска, не блокирует его.
- `X-Content-Type-Options: nosniff`, `X-Frame-Options: DENY`,
  `Referrer-Policy: strict-origin-when-cross-origin`,
  `Permissions-Policy` (камера/микрофон/геолокация/оплата/USB запрещены),
  `Strict-Transport-Security` (без `preload` — это необратимый шаг, требует
  отдельного решения владельца).

## 8. Загрузка файлов

Фото решений — Vercel Blob в режиме **private**: файлы недоступны по прямой
ссылке (проверено — прямой запрос к URL хранилища получает 403). Раздаёт их
только `app/api/solution-photo/[entryId]/route.ts` после проверки владения.
Лимиты: 8 МБ на файл, только `image/*`, сжатие на устройстве ученика перед
отправкой (Vercel отбрасывает запросы больше 4,5 МБ). Импорт JSON — до 2 МБ /
500 задач за раз.

## 9. ИИ-импорт

Ключ AI-сервиса — только в переменной окружения `AI_API_KEY` (никогда в коде
или git). Лимиты входа: 8 МБ файл / 100 000 символов текста, плюс
rate limit 10/час и 40/сутки на пользователя (см. §4) — это платные запросы.
Результат — черновик, который сохраняется в банк только после того, как
репетитор явно отметил и подтвердил каждую задачу (`importTasksJson`, тот же
путь, что у ручного импорта). Модель может ошибаться — предупреждение об этом
есть в интерфейсе.

## 10. Персональные данные и согласия

Карточка ученика (`Student`) хранит минимум: имя/метка, класс, цель, дата
экзамена — без телефона, адреса, паспортных данных. Регистрация требует
явного согласия (`consentAccepted`); для `under16` — дополнительно согласие
родителя (`parentConsentAccepted`), оба энфорсятся на сервере (§3). Политика
конфиденциальности — `/privacy`.

## 11. Секреты

Список переменных окружения (только в Vercel Environment Variables и
локальном `.env`, который в `.gitignore`):

| Переменная | Назначение |
|---|---|
| `DATABASE_URL` | подключение к Neon Postgres |
| `BETTER_AUTH_SECRET` | подпись сессий |
| `BETTER_AUTH_URL` | базовый адрес приложения |
| `AI_API_KEY`, `AI_BASE_URL`, `AI_MODEL` | доступ к LLM для ИИ-импорта |
| `BLOB_READ_WRITE_TOKEN` (или окружение Vercel) | доступ к хранилищу фото |

Grep по репозиторию на паттерны реальных ключей (`sk-`, `postgresql://`,
`AIza`, `vercel_blob_rw`) вне `.env*`/`node_modules` — совпадений нет, только
плейсхолдеры в `.env.example`.

**Ротация:** при подозрении на утечку `BETTER_AUTH_SECRET` — сгенерировать
новый (`openssl rand -base64 32`), заменить в Vercel, передеплоить: все
активные сессии станут невалидны, пользователям придётся войти заново. Для
`DATABASE_URL`/`AI_API_KEY`/`BLOB_READ_WRITE_TOKEN` — перевыпустить в панели
провайдера, заменить в Vercel, передеплоить.

## 12. Резервные копии (Neon)

Neon хранит историю изменений и позволяет восстановление на момент времени
(point-in-time restore) через создание ветки БД из панели: Neon Console →
проект → **Restore** (или **Branches** → «Create branch» с выбором временной
точки). На бесплатном тарифе окно истории ограничено — **проверить точное
значение в Console → Project → Settings → Storage** перед запуском и
зафиксировать здесь. Процедура восстановления: создать ветку на нужный
момент → проверить данные в ней → либо переключить `DATABASE_URL` на новую
ветку, либо выполнить restore основной ветки на это состояние.

Дополнительно, до включения платного тарифа с длинной историей: ручной
`pg_dump "$DATABASE_URL" > backup-$(date +%F).sql` перед каждой миграцией
схемы и раз в неделю; файлы хранить вне репозитория (не в git — там могут
быть персональные данные учеников).

## 13. Реагирование на инциденты

1. Утечка секрета → ротация (§11), передеплой.
2. Подозрение на компрометацию аккаунта → `session.deleteMany({ where: { userId } })`
   через Prisma Studio или скрипт, пользователь входит заново.
3. Повреждение/потеря данных → восстановление из Neon branch или последнего
   `pg_dump` (§12).
4. В любом случае — зафиксировать, что произошло и когда, до исправления.
